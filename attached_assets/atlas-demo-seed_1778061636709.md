# Atlas — Demo Data Seed for `umar.rana@devsinc.com`

## Read this entire document before running anything.

---

## 1. What this is

A one-shot Prisma seed script that augments the existing user `umar.rana@devsinc.com` with realistic demo data for showcasing Atlas to family and friends. The user already exists in the database; this script adds projects, tasks, notes, tables, captures, and supporting entities under that user.

**Augment, not replace.** Existing data for this user is preserved. New seed data appears alongside it. If clean demo state is desired, manually archive existing items before demoing.

**No attachments.** The script doesn't seed attached files (would require R2 uploads). Notes and tasks have no attachments by default. Add manually if needed for visual richness.

**One-shot.** Run once. Re-running will create duplicate seed data. If you need to re-seed cleanly, manually delete previously-seeded entities first.

---

## 2. What gets created

- **5 projects**: 3 professional, 2 personal
- **10 contexts**: standard GTD context list
- **50 tasks**: distributed across projects, dates, and states
- **8 captures**: in Inbox with proposed states (using the new GTD Inbox lifecycle)
- **6 tags**: for tag-filtered demo views
- **14 notes**: across folder hierarchy with all 4 Purpose types
- **4 tables**: across Tables folder hierarchy
- **3 notes designated as Project Briefs** linked to projects
- **Cross-references** between notes and to one table
- **~35 audit log entries** scattered across past 30 days for "lived-in" feel

All dates computed relative to the script run time so the demo is always current.

---

## 3. How to run

Save the script as `scripts/seed-demo.ts` in the Atlas project. Run via:

```bash
npx tsx scripts/seed-demo.ts
```

The script:
1. Looks up the User by email `umar.rana@devsinc.com`
2. Fails clearly if the User doesn't exist
3. Creates seed data under that User's `user_id`
4. Logs a summary of what was created
5. Exits

If the script fails partway through, no data is left in inconsistent state — each entity creation is independent. Re-running creates duplicates of anything that did succeed previously, so prefer fixing the failure cause and starting fresh.

      content: paragraphs.map((p) => ({

        attrs: p.startsWith('## ') ? { level: 2 } : p.startsWith('# ') ? { level: 1 } : undefined,

- Schema field names don't match (e.g., the script uses `folder_id` but the actual field is named differently). Compare against current Prisma schema and adjust field names.
- Required fields not provided (some Tasks/Notes/etc. may have non-nullable fields the script omits). Check error messages for which fields are required.
- Foreign key constraints (referenced entities must exist before referencing them). The script orders creation correctly but verify if errors arise.

**Schema field name notes:**

The script assumes these field names. Adjust if your schema differs:

- Task: `title`, `project_id`, `context_id`, `due_date`, `defer_date`, `completed_at`, `flagged`, `estimated_minutes`, `description`, `is_someday`, `delegated_to_text`, `follow_up_date`, `created_at`
- Project: `name`, `type`, `status`, `folder_id`, `target_date`, `position`, `brief_note_id`
- Note: `title`, `body`, `purpose`, `folder_id`, `project_id`, `is_brief`, `created_at`, `updated_at`
- Capture: `raw_content`, `parser_proposal`, `state`, `created_at`
- Table/TableColumn/TableRow/TableCell: as per Wave 4b schema spec

If field names differ in the actual schema, simple find-and-replace in the script handles the discrepancy.
